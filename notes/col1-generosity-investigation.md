# Col1 (out-of-box) score generosity — root cause

**Date:** 2026-05-06
**Trigger:** Chris noticed the dashboard shows only 2 ones and 0 zeros across all 168 visible score cells, despite the rubric defining a 0–3 range. Investigation revealed the visible distribution is *correct under current data* — but the col1 (OOB) scores themselves are systematically inflated relative to what the rubric expects.

## What the rubric says about col1

`benchmarks/rubric.md` defines col1 as: *"Frontier model, no MCP, bare prompt. The 'where users actually are' baseline."* Expected profile: **Accuracy 1–2, Completeness 1, Citations 0–1, Consumability 2**.

Citations 2–3 across the board, when the rubric expected 0–1.

## Two compounding root causes

### Issue A — col1 is not actually OOB

The "OOB" responses recorded in `data/atoms/*.json` were generated by an agentic Claude system, not by a bare-prompt consumer model:

| scenario | oob_tool field |
|---|---|
| adu-setbacks | Opus 4.7 Adaptive |
| building-records | Opus 4.7 Adaptive |
| cesspool-sewer | Opus 4.7 Adaptive |
| fence-height | Opus 4.7 Adaptive |
| front-porch | Opus 4.7 Adaptive |
| pergola | Claude Opus 4.7 Adaptive |
| pole-barn | ChatGPT 5.5 Thinking |
| tiny-house | Opus 4.7 Adaptive |

Only `pole-barn` used a different vendor. None used "consumer ChatGPT, no tools, bare prompt" — which is what the rubric defines as col1 and what the presentation narrative ("where users actually are") promises.

The opening lines of these "OOB" responses give it away — e.g. adu-setbacks col1 begins:

> "I'll dig into this thoroughly. Setback rules are zone-specific, so I need to pull the actual zoning, overlays, and lot data for your property before answering. Let me check whether I can reach the city's GIS layer first. **Path A is blocked. Let me try web sources to confirm the zoning…**"

That is not a bare-prompt consumer model. That is an agentic system narrating its tool use.

### Issue B — atom-level citation cross-contamination

`data/atoms/*.json` stores claims as atoms with a single `citation` field and a `variants` array listing which column responses support that claim. Most atoms are shared across variants:

| scenario | total oob atoms | oob-only | shared with mcp/enhanced |
|---|---:|---:|---:|
| adu-setbacks | 11 | 1 | **10** |
| building-records | 9 | 2 | 7 |
| cesspool-sewer | 19 | 0 | **19** |
| fence-height | 22 | 1 | **21** |
| front-porch | 20 | 10 | 10 |
| pergola | 14 | 7 | 7 |
| pole-barn | 10 | 3 | 7 |
| tiny-house | 17 | 1 | **16** |

When an atom is shared between variants, the single `citation` field is the *highest-quality* citation any variant produced for that claim. If the enhanced column cited "PCC 33.205.040.C" but the OOB column made the same claim with no citation at all, the merged atom carries `citation="PCC 33.205.040.C"` and `variants=["oob","mcp","enhanced"]`. The aggregator in `skills/atomization/aggregate.py` then credits OOB with an authoritative citation it never produced.

This is why adu-setbacks, fence-height, and cesspool-sewer score `cite=3` at OOB despite the rubric expecting `cite=0–1`.

## Severity

Both issues directly undermine the presentation narrative. The "Col 1 → Col 2 step function on Citations" the rubric calls out as the most important visual cannot land if Col 1 already shows authoritative citations. Either:

- the audience reads it as "OOB is fine, no need for MCP" — wrong conclusion
- the audience notices the flat trajectory and questions the methodology

## Fix options (not yet decided)

1. **Re-run col1 against an actual bare-prompt consumer model** (consumer ChatGPT or Gemini, no tools, no system prompt). This is the rubric-correct fix. Expensive.
2. **Atomization fix: track citations per-variant**, not per-atom. The data model would need an atom field like `citation_by_variant: {oob: null, mcp: "PCC 33.205.040.C", enhanced: "PCC 33.205.040.C"}`. Aggregator pulls the variant-specific citation. Cheap if scoped to citations only.
3. **Documentation fix only**: add a footnote on the dashboard / presentation that col1 represents "Claude Adaptive without MCP/enrichment" rather than "consumer ChatGPT bare prompt." Honest but defeats the rubric's audience-of-real-users framing.

Recommendation: **Issue A wants option 1, Issue B wants option 2.** They're independent. Issue B is the more urgent fix because it silently inflates citations; Issue A is more visible (anyone reading a col1 response will notice it doesn't sound like consumer ChatGPT).

## Issue B fix — applied 2026-05-06

Schema, aggregator, and live server were updated to support per-variant
citations (`atom.citation_by_variant: {oob, mcp, enhanced}`). The legacy
single `citation` field is preserved for display and as a fallback for older
data that doesn't carry the per-variant breakdown.

Files changed:
- `skills/atomization/aggregate.py` — `score_citations` now reads
  `atom_citation_for(atom, variant)` instead of `atom.citation`.
- `skills/atomization/atomize.py` — `ATOM_SCHEMA` accepts the optional
  `citation_by_variant` object.
- `skills/atomization/prompt.txt` — instructs the evaluator to emit
  per-variant citations and explicitly forbids inheriting another variant's
  citation.
- `skills/atomization/backfill_citation_by_variant.py` — one-shot script
  that backfills `citation_by_variant` on the existing 8 atoms files using
  a substring heuristic (PCC/Title/ORS/portland.gov tokens, plus
  PortlandMaps with both spaced and concatenated forms).
- `server.py` — `_derive_scores_from_atoms` (the live overlay used by
  `/api/three-column-state`) and `_unique_citations_for` (the per-column
  citation chip list) now both prefer per-variant citations.

Backfill outcome across the 8 atoms files: 274 atoms; 230 (atom, variant)
pairs kept their citation, 97 were cleared because the cited token did not
appear in that variant's response text, 98 atoms had `citation = null` to
begin with.

### Outstanding (Issue A — not addressed)

Col1 was generated by the same agentic Claude system as col2/col3, not by
a bare-prompt consumer model. The rubric's "where users actually are"
narrative still requires re-running col1 against a real OOB tool. That
remains open.
